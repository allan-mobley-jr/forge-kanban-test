# Application Specification
> Generated by Forge /plan on 2026-03-09

## Purpose
KanbanFlow is a minimal, client-side personal task management kanban board. It targets a single user who wants a fast, keyboard-driven way to organize tasks without authentication, accounts, or server infrastructure. All data lives in the browser via localStorage, making it instantly available and offline-capable after the first page load.

The core value proposition is simplicity: three columns (To Do, In Progress, Done), drag-and-drop cards, keyboard shortcuts, and nothing else. No collaboration features, no integrations, no settings pages.

## Architecture
- **Single-page application**: One route (`/`) renders the entire board. No dynamic routes, no API routes.
- **Route structure**: `src/app/layout.tsx` (Server Component, metadata, provider wrapper) → `src/app/page.tsx` (Server Component, renders Board) → `Board` (Client Component, entry point for all interactivity).
- **Component hierarchy**: Board → Column(×3) → Card(×N) + AddCardButton. Modal overlay: CardDetailModal (with MarkdownEditor, ColorLabelPicker, DeleteConfirmation). Search overlay: SearchDialog.
- **State management**: React Context (`BoardContext`) with `useReducer` for all board mutations (ADD_CARD, UPDATE_CARD, DELETE_CARD, MOVE_CARD, REORDER_CARD). No external state library needed.
- **Data persistence**: Custom `useLocalStorage` hook wrapping `localStorage.getItem`/`setItem` with JSON serialization, SSR safety, and corruption recovery. Auto-save on every dispatch.
- **Data shape**: `Board { columns: Column[], cards: Record<string, Card> }`. Cards stored in a flat map, columns hold `cardIds` arrays for ordering.
- **Server vs Client**: Layout and page shells are Server Components (for metadata and streaming). The Board component and everything below it are Client Components (`'use client'`), since the entire UI is interactive with no server-side data.

## Technology Stack
- **Framework**: Next.js 15 with App Router, TypeScript, Tailwind CSS (all from bootstrap)
- **Drag-and-drop**: `@dnd-kit/core` + `@dnd-kit/sortable` + `@dnd-kit/utilities` — actively maintained, accessible, keyboard-friendly, TypeScript-native
- **Markdown rendering**: `react-markdown` — safe by default (no raw HTML), React-native rendering
- **Component primitives**: shadcn/ui (Dialog, AlertDialog, Button, Command) — accessible, built on Radix, Tailwind-native
- **Typography**: `@tailwindcss/typography` — prose styling for rendered markdown
- **Card IDs**: `crypto.randomUUID()` — zero dependencies, supported in all modern browsers
- **Testing**: vitest + @testing-library/react (unit), Playwright (e2e) — provided by Forge bootstrap
- **No backend**: No database, no auth, no API routes, no environment variables
- **No additional vendor skills needed** beyond defaults

## Design System
- **Layout**: Minimal app shell — slim header with title + search trigger, three-column horizontal flex board filling remaining viewport height
- **Responsive**: Desktop-first. Desktop (lg: 1024px+): columns side by side. Tablet/mobile (< 1024px): horizontal scroll, each column min-w-[280px]
- **Typography**: System font stack (font-sans). Scale: xl/bold (app title), sm/semibold/uppercase (column headers), sm/medium (card titles), xs/muted (captions, dates)
- **Colors**: Slate palette (bg-slate-50 board, bg-slate-100 columns, bg-white cards). Blue-600 primary actions, red-600 destructive. 5 label colors: red, yellow, green, blue, purple (Tailwind defaults)
- **Spacing**: 4px rhythm. Column gap-4, card gap-2, card padding p-3, modal padding p-6
- **Icons**: Lucide React (bundled with shadcn/ui) — Plus, X, Trash2, Search, Calendar
- **Drag visual**: shadow-lg on drag overlay, opacity change on source card
- **No dark mode**: Not in requirements. Light-only keeps design surface manageable.

## Constraints & Risks
- **Drag-and-drop complexity**: Highest-risk feature. @dnd-kit's sortable-across-containers pattern requires careful setup. Click-vs-drag distinction needed (pointer sensor with 5px activation distance). Touch support via TouchSensor with 250ms delay.
- **localStorage limits**: ~5 MB browser quota. Sufficient for personal task management. Schema versioning included for future migration. Corruption recovery falls back to empty board with error prompt.
- **Keyboard shortcut conflicts**: Single-key shortcuts (N, 1, 2, 3) must be guarded against firing in text inputs. Cmd+K may conflict with browser shortcuts — use preventDefault().
- **Markdown bundle size**: react-markdown + remark ecosystem adds ~80-100 KB gzipped. Acceptable for this app. Dynamic import possible if bundle size becomes a concern.
- **SSR hydration**: localStorage unavailable during server render. useLocalStorage hook returns defaults during SSR, hydrates on mount. No flash of empty state — server renders the default empty board, client hydrates with saved data.
- **Offline support**: Requires service worker / PWA configuration (next-pwa or custom). Not provided by Next.js out of the box.
- **Subjective design criteria**: "Clean whitespace, no visual clutter" cannot be automatically verified. Rely on defined spacing rhythm and color palette for consistency.

## Decisions Log
- **DnD library**: Chose @dnd-kit over @hello-pangea/dnd — @dnd-kit is actively maintained with better accessibility support and modern React patterns. @hello-pangea/dnd is a fork of the archived react-beautiful-dnd.
- **Component library**: Chose shadcn/ui selectively (Dialog, AlertDialog, Button, Command) over building all custom. These components handle complex accessibility (focus trapping, keyboard navigation) that is error-prone to implement from scratch. Simple components (cards, columns, labels) are built custom with Tailwind.
- **Card IDs**: Chose crypto.randomUUID() over uuid package — zero dependencies, modern browser support, no @types package needed.
- **Focus trapping**: Handled by Radix Dialog (via shadcn) — no separate focus-trap-react dependency needed.
- **State management**: Chose React Context + useReducer over Zustand/Redux — sufficient for single-user localStorage app, no external dependency.
- **Issue decomposition split**: Split data model issue into types/localStorage hook (issue #3) and BoardContext (issue #4) per advocate review — keeps PRs right-sized.
- **Offline support**: Added as explicit issue (#14) per advocate review — PROMPT.md explicitly requires "offline-capable" which needs PWA/service worker configuration.
