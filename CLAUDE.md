### State Machine

All project state is encoded in GitHub Issue labels. Only one issue is active at a time â€” the agent works on the lowest-numbered open issue.

| Label | Meaning |
|-------|---------|
| `agent:in-progress` | An agent is actively working on this issue |
| `agent:done` | PR opened, awaiting CI and auto-merge |
| `agent:needs-human` | Blocked on a decision only a human can make |

- **No `agent:*` label** = backlog (unclaimed, ready to build)
- `ai-generated` = metadata label marking agent-filed issues/PRs

`/sync` detects stale `agent:in-progress` labels and recovers them. It also detects `agent:done` issues whose PR has `CHANGES_REQUESTED` and reports them so `/forge` routes to `/revise`. Issue ordering is the dependency graph â€” lower-numbered issues are built first.

### Filing Issues

To have the agent work on a human-filed issue, just create it on GitHub. The agent picks up the lowest-numbered open issue without an `agent:*` label. No special labels needed â€” just create the issue and the agent will get to it in order.

### Resumability

There is no local state to lose. If a session ends mid-work:

- GitHub Issues and labels reflect the last known state
- Any in-progress branch has been pushed (or can be abandoned)
- Starting a new `claude` session and running `/forge` picks up exactly where it stopped
- `/sync` detects stale `agent:in-progress` labels and recovers them

### Context Management

After `/plan` completes, use `/clear` before starting the build loop â€” `/sync` will re-establish all necessary context from GitHub. Do not `/clear` between individual `/build` cycles. Sub-agents spawned via Task have their own context windows and do not consume the main session's.

### Autonomous Operation

For fully autonomous headless operation, use `forge run` from the project root. This wraps `claude -p "/forge"` in a restart loop that survives context exhaustion and picks up where it left off. Each session writes `.forge-temp/status.json` for progress tracking and `.forge-temp/exit-status` for the restart loop.

## Workflow

- GitHub Issues are the task queue. No issue means no work.
- Every feature is implemented on a feature branch and submitted as a PR.
- PRs require CI to pass (lint + typecheck + test + build) before merge.
- PRs are auto-merged immediately after CI passes.- All project state lives in GitHub â€” nothing important is stored locally.

### PR Merge

**Mode:** auto â€” PRs are merged as soon as CI passes.

No code review step. The agent opens a PR, waits for CI, and squash-merges immediately. Human `CHANGES_REQUESTED` reviews are still routed to `/revise` if submitted before the merge.

### Production Deployment

The `production` branch is off-limits. Do not push to it, create PRs targeting it, or modify it in any way.
