## Forge System

This project is managed by the **Forge** autonomous development system. The main Claude Code session acts as the orchestrator, routing work through skills and spawning specialized sub-agents via the Task tool.

- Skills (`/forge`, `/plan`, `/build`, `/revise`, `/sync`, `/ask`) define the orchestration workflow.
- Sub-agents (spawned via Task) handle research, review, testing, and debugging.
- GitHub Issues and PRs are the sole state machine тАФ no local state files.

If you are the main session, run `/forge` to begin. If you were spawned as a sub-agent, follow your task prompt тАФ the rest of this document is shared project context.

### Architecture

```
Main session тЖТ /forge (orchestrator skill)
  тФЬтФАтФА /sync   тАФ reads GitHub issue/PR state, returns structured summary
  тФЬтФАтФА /plan   тАФ spawns 4 research sub-agents in parallel, files GitHub Issues
  тФЬтФАтФА /build  тАФ claims one issue, implements, spawns review+test+debug sub-agents, opens PR
  тФЬтФАтФА /revise тАФ addresses PR review comments, pushes fixes to existing branch
  тФФтФАтФА /ask    тАФ escalates blocking questions to human via GitHub comments
```

The loop: `/forge` тЖТ `/sync` тЖТ route to `/plan`, `/build`, or `/revise` тЖТ back to `/forge` тЖТ repeat until all issues are closed or a human decision is needed.

### State Machine

All project state is encoded in GitHub Issue labels. Only one issue is active at a time тАФ the agent works on the lowest-numbered open issue.

| Label | Meaning |
|-------|---------|
| `agent:in-progress` | An agent is actively working on this issue |
| `agent:done` | PR opened, awaiting human review and merge |
| `agent:needs-human` | Blocked on a decision only a human can make |

- **No `agent:*` label** = backlog (unclaimed, ready to build)
- `ai-generated` = metadata label marking agent-filed issues/PRs

`/sync` detects stale `agent:in-progress` labels and recovers them. It also detects `agent:done` issues whose PR has `CHANGES_REQUESTED` and reports them so `/forge` routes to `/revise`. Issue ordering is the dependency graph тАФ lower-numbered issues are built first.

### Filing Issues

To have the agent work on a human-filed issue, just create it on GitHub. The agent picks up the lowest-numbered open issue without an `agent:*` label. No special labels needed тАФ just create the issue and the agent will get to it in order.

### Resumability

There is no local state to lose. If a session ends mid-work:

- GitHub Issues and labels reflect the last known state
- Any in-progress branch has been pushed (or can be abandoned)
- Starting a new `claude` session and running `/forge` picks up exactly where it stopped
- `/sync` detects stale `agent:in-progress` labels and recovers them

### Context Management

After `/plan` completes, use `/clear` before starting the build loop тАФ `/sync` will re-establish all necessary context from GitHub. Do not `/clear` between individual `/build` cycles. Sub-agents spawned via Task have their own context windows and do not consume the main session's.

### Autonomous Operation

For fully autonomous headless operation, use `forge run` from the project root. This wraps `claude -p "/forge"` in a restart loop that survives context exhaustion and picks up where it left off. Each session writes `.forge-temp/status.json` for progress tracking and `.forge-temp/exit-status` for the restart loop.

## Workflow

- GitHub Issues are the task queue. No issue means no work.
- Every feature is implemented on a feature branch and submitted as a PR.
- PRs require CI to pass (lint + typecheck + test + build) before merge.
- The human approves every PR before it's merged.
- All project state lives in GitHub тАФ nothing important is stored locally.

## Conventions

- Use Server Components by default. Only add `'use client'` when interactivity is needed.
- Fetch data in Server Components, not in client-side effects.
- Use Tailwind CSS for styling. No custom CSS files unless absolutely necessary.
- Follow the existing file and component naming patterns.
- Consult AGENTS.md for Next.js framework patterns (Server Components, Server Actions, streaming, caching).
- Use `agent-browser` to visually verify UI changes before submitting PRs.
- Every commit references the issue it closes: `feat: title (closes #N)`
- Co-locate unit tests with source: `Component.tsx` -> `Component.test.tsx`
- E2E tests go in the `e2e/` directory (Playwright, chromium only in CI)

## Security

- Do not push directly to main тАФ all changes go through feature branches and PRs.
- Do not force-push (`--force`, `-f`) тАФ it rewrites shared history.
- Do not run destructive GitHub commands (`gh repo delete`, etc.).
- Do not read, print, or log environment variables or secrets.
- Do not modify CI workflows (`.github/workflows/`) тАФ these are managed by Forge.
- Protected files/dirs are enforced by hooks: `.env*`, `.git/`, `CLAUDE.md`, `.claude/skills/`, `pnpm-lock.yaml`, `.vercel/`, `node_modules/`, and `.github/workflows/` (write-only). `PROMPT.md` and `SPECIFICATION.md` become protected after the initial planning phase.

## Do Not

- Write to `.env` files directly тАФ use `vercel env` for environment variables.
- Modify `pnpm-lock.yaml` manually тАФ let `pnpm install` handle it.
- Skip quality checks тАФ always run `pnpm lint && pnpm tsc --noEmit && pnpm test && pnpm build`.
- Work on code without a corresponding GitHub Issue.
- Merge PRs without human approval.
- Modify `CLAUDE.md` or files in `.claude/skills/`, or modify `PROMPT.md` or `SPECIFICATION.md` outside the `/plan` archival step (protected after initial planning).
